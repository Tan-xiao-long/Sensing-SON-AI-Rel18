## RAN3#120 (2023.05.22–26, 仁川)

### 本次会议 SON/MDT 主要共识与增强内容

**SHR/SPR**
- 将此前 WA 转为正式共识：SPR 触发门限以百分比取值表示（与 SHR 一致）。
- 达成一致：若 SN 经由 SN RRCReconfigurationComplete 收到 SPR 可用指示，SN 应通知 MN UE 侧有 SPR 可取，如在 S-NODE MODIFICATION REQUIRED 消息中新增 IE。
- 通过多份 TP（38.300/37.340/38.413/38.423/38.473 BLCR 的 TP，R3-233379、R3-233399、R3-233408），落实 inter-RAT SHR 与 SPR 的 stage2/stage3 文本。
- 发出 LS 至 RAN2（R3-233380），涉及 inter-RAT SHR 与 SPR 的 UE 上报内容与机制确认。
- 遗留开放问题：SHR/SPR 的 UE 上下文检索方案（C-RNTI 方案 vs Mobility Information 方案）、MN 发起 PSCell change 时由哪个节点决定 T310/T312 触发、inter-RAT SHR 与 RLF Report 的关联方案，留待后续会议。

**MRO**
- 快速 MCG 恢复：同意在 R18 定义 Case c（新场景），解决方案后续讨论。
- 语音回落（voice fallback）：确认 RAN2 已决定在 LTE RLF report 中引入 voice fallback 指示；RAN3 达成 WA——在 S1/NG 的 Inter-system HO Report 中定义新的 handover report type。
- 通过 TP：R3-233396（36.300/38.300/38.413 MRO 增强）、R3-233397（voice fallback 的 38.413/38.300/36.300 stage2/3 文本），并请报告人向 RAN 全会更新受影响规范列表（含 TS 36.300）。
- 遗留：CPAC 候选 PSCell 列表是否发给 source/last serving SN、CPAC UHI、pre-R18 UE 支持问题。

**RACH Enhancements**
- 正式引入 RACH Indication：通过 38.470 TP（新增自优化支持过程，gNB-DU→gNB-CU 指示 RACH 发生，R3-233498）、36.423 TP（X2 上的 RA Report Indication，R3-233499）及 36.300 中 RA report 命名更新（R3-233450）。
- 通过 38.423 TP（R3-233500）：在 Xn 的 ACCESS AND MOBILITY INDICATION 中，除 RACH Report Container 外可选携带 "PSCell list"。
- 明确不考虑经 NG/S1 转发 RA report。
- 新分配 BLCR：38.420 由 Qualcomm、38.470 由 CMCC 负责。
- FFS：取回 RA Report 的 MN 与 PSCell 相关 SN 无连接时的转发处理。

**NPN**
- Stage3 细节确定：maxnoofCAGforMDT 取值范围定为 256；新增独立的 SNPN Cell Based MDT 与 SNPN TAI Based MDT 区域范围；支持 multi-SNPN MDT area scope；确认无需在接口上传递 SNPN 的 user consent。
- 通过 TP：R3-233470（38.413 MDT area scope 支持 NPN）、R3-233452（38.413/38.423 NPN 遗留问题）。
- SNPN-wide area scope 是否需要仍未定论。

**SON for NR-U**
- 通过 38.423 TP（R3-233491）：XnAP RESOURCE STATUS UPDATE 中的 EDT（Energy Detection Threshold）UL 取值为所配置最大 EDT UL 的平均值。
- 明确 UE 无需在 RLF report 中上报所应用 EDT UL 的平均值。
- WA：在 XnAP 与 F1AP 的 RESOURCE STATUS UPDATE 中引入可选的 per NR-U Channel 无线资源状态负载度量（Radio Resource Status per NR-U Channel）。
- FFS：F1 接口是否引入 EDT UL 上报。

**MDT Enhancements**
- 达成结论：不要求 NR 信令 logged MDT 优先级低于 E-UTRAN 信令 logged MDT。
- 对在 NGAP 中引入 cause value（用于告知核心网/OAM NR 信令 logged MDT 因 LTE 信令 logged MDT 保护而失败）未达成共识，不引入。

### 本次会议 AI/ML 主要共识与增强内容

**Stage2**
- 通过新过程的 stage2 文本 TP（R3-233342，TS 38.300 BLCR），并确定 stage2 撰写方式：先描述系统行为，待方案稳定后再按用例补充描述。
- 通过 38.423 TP（R3-233509）：在 AI/ML INFORMATION REQUEST 中引入 Requested Prediction Time IE。
- 明确 Requested prediction time 为"合理未来内的特定时间点"，且该时间点配置对 one-time 与 periodic 两种上报方式均适用。
- 编码方式待在 INTEGER 与 ENUMERATED 两选项间下选；"预测覆盖一段时间区间（time interval）"的必要性 FFS。

**LB（负载均衡）**
- 明确 R18 不支持节点级/小区级的 UE performance 收集（UE performance 仅作为 HO 后的 feedback）。
- 通过部分上报（partial reporting）TP（R3-233512）：被请求节点若只能提供部分信息，在 AI/ML INFORMATION RESPONSE 中携带 Node Measurement Initiation Result IE 和/或 Per Cell Measurement Initiation Result IE。
- Failed Report Characteristics 拆分为两个字段：per cell 测量失败特征（小区粒度）与 per node 测量失败特征（节点粒度），并为两者各引入 Cause IE；确认复用遗留 cause 值 "Measurement not Supported For The Object" 与 "Measurement Temporarily not Available"，是否新增其他 cause 值 FFS。
- 确定 UE Associated Info Result Item 中应包含 Source NG-RAN node UE XnAP ID reference IE（在 update 消息中）。
- 明确 UE Performance Feedback 上报的时间配置至少包含：测量收集时长（从 HO 执行起、含最后一次上报）、时长内周期或一次性上报、周期上报时的周期取值；具体信令方式后续讨论。
- 通过 R3-233513（38.423/38.420 TP）：在 AI/ML INFORMATION UPDATE 中仅新增 UE trajectory IE。

**ME（移动性优化）**
- 通过小区级 UE 轨迹预测交换 TP（R3-233112，38.423）。
- 确定预测 UE 轨迹中"预测驻留时间（Predicted Time UE Stays in Cell）"的 presence 为 Optional。
- 达成关键结论：R18 中 UE 轨迹预测仅限下一跳（one hop）目标 NG-RAN 节点；源节点可在切换后从目标节点收集该 UE 在单一目标节点内一个或多个小区的实测 UE 轨迹（UE History Information 形式）。
- 以 UE performance feedback 上报过程为基线，支持源 gNB 的实测 UE 轨迹收集；请求方式细节 FFS。

**ES（节能）**
- 发出 LS 至 SA5（R3-233515）：请求 NG-RAN 节点配置统一规则，将节点 Energy Cost（EC）值映射/归一化到能耗测量。
- 确定 EC 编码为索引（index）：由 OAM 归一化并定义，取值为 0 至最大值的整数，最大值需保证足够精度。
- FFS：R18 是否引入 inferred EC 以及 "additional load" 描述。

**其他接口（NG/F1/E1）**
- 本次无实质新共识（仅有 E1/F1 影响的讨论文稿，维持"先完成非拆分架构再讨论拆分架构"的原则）。

**Others（AI/ML 用 MDT 增强）**
- 达成共同理解：信令 MDT 场景下 TCE 可通过 UE 标识识别 UE；管理 MDT 因匿名化设计，TCE 无法唯一识别 UE。
- 后续评估方向：连续 MDT 在 idle→active 转换期间的数据间隙可接受度；核查 m-based MDT 标识（IMEI-TAC、NR-CGI、TR/TRSR）能否支持 TCE 关联 logged MDT 与 immediate MDT。
