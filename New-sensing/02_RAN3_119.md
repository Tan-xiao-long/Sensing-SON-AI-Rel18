## RAN3#119 (2023.02.27–03.03, 雅典)

### 本次会议 SON/MDT 主要共识与增强内容

**SHR/SPR**
- 同意 LTE→NR 方向的 inter-RAT SHR：RAN3 认为在对 LTE 无影响的前提下支持基于 T304 触发的 LTE→NR SHR 有收益，并发 LS（R3-230845/R3-232140 前身）请 RAN2 考虑并反馈。
- inter-RAT SHR 转发机制确定 Option 3 为基线：接收节点将 SHR 转发给产生该 SHR 触发条件的对应节点。
- WA：LTE→NR inter-RAT SHR 内容至少包含 Source LTE cell 和 Target NR cell；来自 UE 的更多信息待定（取决于关联/UE 上下文检索机制）。
- SPR 触发/根因归属确定：classic CPA 及 SN/MN 发起的 classic PSCell change/CPC 场景下，target SN 决定 T304 触发并做根因分析；intra-SN CPC 场景下，source SN 决定 T310/T312 触发并做根因分析。
- SPR 目的明确：若触发为 T310/T312，SPR 用于优化 source PSCell 底层问题及移动性中的 PSCell change 配置；T304 触发的目的（移动性配置 vs RACH 问题）FFS。
- MN 发起的 CPC 场景：若触发为 T310/T312，至少下发 SPR 配置的 MN 参与 SPR 优化；由哪个节点决定触发/做根因留有 3 个 Option 待下次会议下选。

**MRO**
- Stage2 TP（R3-230993，38.300）agreed：引入语音回落（voice fallback）inter-system 切换失败的失败类型定义与检测描述章节。
- CPAC MRO：若为 CPA/CPC 配置了多个事件，UE 上报首个触发的 CPAC 事件以及两个触发事件间的时长。
- "wrong PSCell change/addition" 定义拆分为两个子情形：错误候选小区来自发起节点提供的列表，或由目标节点选择。
- SCGFailureInformation 增强原则：网络节点已有的信息不放入该消息；复用 R17 信令机制经 Xn 由 MN 向 source SN/last serving SN 上报 CPA/CPC 失败相关信息。
- fast MCG recovery：UE 至少上报发生 SCG failure 的 PSCell、fast MCG recovery 失败原因（至少 T316 expiry、SCG failure、SCG deactivated 等）及 SCG failure 类型（t310-Expiry、randomAccessProblem、rlc-MaxNumRetx）；相应 LS 发 RAN2（R3-230992）。

**RACH Enhancements**
- 确定 F1 上的 RACH Indication 采用 non-UE associated 信令，并以 Xn 接口信令设计为 F1 stage3 CR 起点。
- RACH report retrieval 的 Stage2 TP agreed：38.300（R3-231040）与 38.401（R3-230996），gNB-DU 向 gNB-CU 指示 CU 未知的成功 RACH（如 BFR 触发）；RA report 命名待下次会议澄清。
- 面向 RAN2 的 RACH 优化 LS（含 feature priority、RACH partition 配置、时间戳、网络控制上报 4 个选项）本次仅 noted，留待下次会议。

**NPN**
- Stage3 方案下选：以 "在 MDT Area Scope 现有 CHOICE 结构内外均增加 CAG list" 的 Option 1 为基线，排除仅在结构外增加 PNI-NPN 区域的 Option 2。
- NGAP TP（R3-231022，38.413）agreed，引入 PNI-NPN Area Scope for MDT 新 IE（置于 tabular 末尾，附兼容性 Editor's note）。
- SNPN Area Scope 编码方式 FFS；NPN 用户同意（user consent）问题继续等待 SA3 回复。

**SON for NR-U**
- MLB：确定 DU 在 CU 请求时通过 Resource Status Reporting 过程向 CU 上报 UL COT，DU 如何获取该信息由实现决定。
- F1 接口 UL COT 上报粒度确定为 NR-U channel 级；对应 38.473 TP（R3-230980）agreed。
- MRO 方向（RLF report 增加 LBT 等待时间、UL EDT 等）本次仅讨论，无新结论。

**MDT Enhancements**
- 本次无实质新共识：NR-DC 下 signalling based immediate MDT 的 TP 因范围争议仅 noted，cross-RAT logged MDT 继续等待 RAN2 进展。

### 本次会议 AI/ML 主要共识与增强内容

**Stage2（12.2.1）**
- 将 WA 升级为正式 agreement：Rel-18 不考虑 NG 接口上的 AI/ML 信息交换方案（仅聚焦 Xn）。
- "Procedures shall be data type agnostic" 的 WA 本次讨论其含义（是否指示 input/output/feedback 用途）未定，留待 stage3 讨论后回看。
- 38.300/38.401/38.423 的 AI/ML BLCR 均 endorsed（含 split 架构下 AI/ML-RAN 功能的 38.401 CR）。

**LB（12.2.2.1）**
- Xn 影响 TP（R3-230978，38.423/38.420）agreed。
- 确定已同意的 Class 1 过程（AI/ML INFORMATION REQUEST/RESPONSE，名称待定）用于配置 UE performance feedback 上报。
- 在新的 REQUEST 消息中引入"切换事件后提供 UE performance feedback"的指示（显式/隐式方式 FFS）；同时在 HO REQUEST 消息中引入触发指示，表明 HO 完成后需要 UE performance feedback。
- 确定已同意的 Class 2 non-UE associated 过程（AI/ML INFORMATION UPDATE，名称 FFS）用于 UE performance feedback 上报。
- 同意在 AI/ML 信息过程（REQUEST/RESPONSE/UPDATE）中支持 partial reporting 机制，具体方案后续讨论。

**ME/移动性（12.2.2.2）**
- Cell-based UE Trajectory Prediction 交换的 38.423 TP（R3-230855）agreed（结构同 UHI，经 HO REQUEST 传给目标 gNB）。
- 确定本 release 预测 UE Trajectory 中不引入 predicted RRC state。
- 确定本 release 预测 UE Trajectory 中不引入 beam index 信息。
- "Predicted Time UE Stays in Cell" IE 的存在性（O/M）FFS；预测轨迹是否跨多个 NG-RAN 节点 FFS。

**ES/节能（12.2.2.3）**
- 关键下选：引入 Energy Cost（EC）作为 gNB 间经 Xn 接口共享的 AI/ML 节能度量（替代直接交换 EE 的各方案）。
- 采纳 Option-3a：EC 按请求（upon request）在 Xn 接口交换。
- EC 定义：可以是与附加负载（additional load）相关的推断能耗，或邻节点针对附加/当前负载的实际能耗值（细节待定）；EC 为 gNB 级取值。

**其他接口（12.2.2.4）**
- 本次无实质新共识：无 F1/E1/NG 相关结论，维持"先非拆分架构、后拆分架构"的既有原则。

**Others / MDT 增强（12.3）**
- WA：使 UE 跨 RRC 状态连续采集 MDT 测量并向网络提供连续时间序列数据用于 AI/ML Training/Retraining 是有收益的；R18 中收益 vs 影响仍需评估。
- 实现 Continuous MDT 的具体方案 FFS；给 SA5 的 draft LS（R3-230976）本次仅 noted 未发出。
